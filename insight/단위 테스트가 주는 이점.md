### ✅  1. 코드가 올바르게 동작하는지 빠르게 확인 가능

- 여러개의 코드가 뭉쳐있을 때 단위 테스트를 만들지 않는다면 그 코드들을 테스트 하기 위해서는 전체를 한 번에 엮어서 테스트해야한다.
- 단위 테스트를 할 수 있는 구조이고, 특정한 로직만 단위 테스트로 검증할 수 있다면 **일부만 아주 빠르게 실행**할 수 있다.
- 즉 내가 만든 코드를 **빠르게 확인**할 수 있다.

### ✅  2. 디버깅이 쉬워짐

- 어떤 문제가 발생했을 때 디버깅이 쉬워진다.
- **문제 범위를 한 클래스나 메서드로 좁힐** 수 있다.
- 이를 단위 테스트로 검증을 한다면 **확인할 코드도 줄어들**고,
- **원인을 빠르게 찾는 데 도움**이 된다.

### ✅  3. 개발 속도 향상에 도움

- 처음에는 당연히 단위 테스트 코드 작성에 시간이 걸린다.
- 하지만 이후 반복적으로 발생하는 테스트 시간을 줄여두고 문제 원인을 찾는 시간도 줄여준다.
- 이를 장기적으로 보면 개발 속도를 향상시켜주는데 도움이 된다.

### ✅  4. 품질을 높여줌

**개별 클래스/메서드에 대한 빠른 검증 → 조기에 오류를 발견할 수 있게 도와 줌**

- 즉 문제에 대해 개발 서버에 배포한 뒤에 발견하는 것 보다 배포 전에 발견되는 가능성이 증가한다. → 조기에 오류를 발견할 수 있다는 것은 초기에 품질을 높일 수 있다는 뜻이 된다.

**검증 코드가 증가하면 → 상대적으로 결함이 줄어 들 가능성이 높아짐**

- 결과적으로 단위 테스트가 많아지고 단위 테스트가 검증할 대상이 많아질 수록 품질이 높아진다.

### ✅  5. 코드 이해에 도움

- 테스트 코드에서의 입력, 실행, 검증 → 코드 동작의 이해에 도움 + 코드만 보는 것보다 나음
- 일종의 문서로서의 역할 → 단위테스트들이 쌓이면 이 메소드 혹은 클래스가 실행하면 결과가 이렇다는 것을 코드상으로 확인할 수 있으므로

### ✅ 그 외…

**자동화된 테스트로 활용**

- 배포 과정에서 자동화된 테스트로 활용할 수 있게 되고, 잘못된 코드가 배포될 가능성이 줄어든다.

**안정적인 리팩토링 지원**

- 단위 테스트가 있기 때문에 안정적으로 코드 수정이 가능하다.

**테스트 가능한 구조 유도**

- 단위 테스트를 만들기위해서는 코드가 테스트 가능한 구조여야한다.
- 따라서 자연스럽게 테스트 가능한 구조가 유도가 된다.

이 외에도 많은 이점들이 있다. 정리하자면 TDD를 하지 않더라도 단위 테스트 코드를 만든다는 것은 만드는 것만으로도 위와 같은 여러가지 이점들이 생기기 때문에 단위 테스트 작성을 하는 것이 좋다.
